# ML risk prioritization — assumptions and limitations

## Problem statement

We train a **binary classifier** that estimates the probability that a **vulnerability finding should be prioritized for remediation soon** under organizational policy. This is **not** prediction of real-world exploitation or breach likelihood.

## Target (label) definition

Labels are **proxy targets** generated by `compute_proxy_label()` in `ml/features/engineering.py`:

- `1` (urgent): CRITICAL severity; or known exploit with CVSS ≥ 7; or overdue active finding; or high asset criticality with HIGH/CRITICAL severity; or exploit with HIGH severity.
- `0`: otherwise.

These rules are intentionally aligned with common SOC prioritization heuristics so the model learns a **ranking function** correlated with policy, not ground-truth outcomes.

## Data leakage controls

- We **do not** use `internal_priority_score` as a feature (it would trivially encode analyst judgment and may be missing).
- `due_at` is transformed into **`days_until_due`** relative to inference reference time (negative = overdue). This is schedule context, not a direct copy of the label rule in feature form for all cases, but **highly correlated** with proxy labels when overdue — acceptable for an internal triage assistant, **not** for external compliance claims.

## Features

| Group | Fields |
|--------|--------|
| CVE / exposure | `cvss`, `epss`, `exploit` |
| Asset | `asset_criticality`, `asset_name`/`asset_type` compressed to `text_n_tokens` |
| Finding workflow | `days_open`, `days_until_due`, `has_assignee`, `status_str` |
| Taxonomy | `severity_str` (one-hot after preprocessing) |

**NLP:** We use a lightweight numeric proxy `text_n_tokens` (token count on concatenated title + asset name + type). Full TF–IDF or embeddings are a documented upgrade path.

## Model

- **Algorithm:** `HistGradientBoostingClassifier` inside a `ColumnTransformer` pipeline (median imputation for numerics, one-hot for low-cardinality categoricals).
- **Calibration:** Not calibrated in v1; scores are **raw probabilities** from the ensemble — interpret as **relative** risk, not calibrated P(exploit).

## Operational notes

- Retrain when schema or label policy changes; bump `MODEL_VERSION` in `train_prioritizer.py`.
- Class imbalance: hold-out metrics are reported in bundle metadata; use **PR-AUC** when positives are rare.
